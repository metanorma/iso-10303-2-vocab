## Project Overview

ISO 10303-2 Vocabulary website — a concept browser for STEP (Product Data Representation and Exchange) terminology. Built with the [Glossarist Concept Browser](https://github.com/geolexista/glossarist-vocabulary-browser) (`@glossarist/concept-browser`), deployed to GitHub Pages at https://metanorma.github.io/iso-10303-2-vocab/.

This information service is provided by **ISO/TC 184/SC 4/TF 1**, the task force that manages the ISO 10303-2 vocabulary.

Two datasets in `.datasets/`:
- **iso10303-2-terms** — 685 term definitions from Clause 3 of ISO 10303-2
- **iso10303-2-express** — 7,363 EXPRESS entity concepts from application modules and integrated resources

## Commands

```sh
# Install dependencies
npm install

# Local dev server (generates data + starts Vite at localhost:5173)
npm run dev

# Full build (generate + edges + favicons + vite build)
npm run build

# Preview production build
npm run preview

# Regenerate concepts from private iso-10303 source (requires suma gem + ISO_10303_SRC)
ISO_10303_SRC=/path/to/iso-10303 rake concepts:all    # express + terms
ISO_10303_SRC=/path/to/iso-10303 rake concepts:express  # EXPRESS only
ISO_10303_SRC=/path/to/iso-10303 rake concepts:terms    # terms only
```

**Important:** The `vite` binary lives inside the concept-browser's nested `node_modules`. The `dev` and `preview` scripts use the explicit path `node_modules/@glossarist/concept-browser/node_modules/.bin/vite` to avoid resolving to the system Ruby `vite` gem.

## Architecture

This is a **deployment configuration repository** — not a traditional application. There is no application source code; the entire site is generated by the `@glossarist/concept-browser` npm package.

**Data flow:**
1. Concept YAML files in `.datasets/` are the canonical data source (committed to git)
2. `concept-browser build` runs generate, edges, favicons, and Vite build in one step
3. `site-config.yml` controls identity, datasets, branding, navigation, and features
4. About pages: `about.md` (site-level), `datasets/<id>/about.md` (per-dataset, auto-discovered via `localPath`)
5. Output goes to `dist/`, deployed to GitHub Pages

**Directory layout:**
- `datasets/` — local source directories with about files (mirrors oiml-vocab pattern)
- `.datasets/` — committed concept data (YAML concepts, register.yaml, metadata.yaml)
- `site-config.yml` — site configuration (datasets, branding, pages)
- `Rakefile` — concept extraction from the private `iso-10303` source repo via the `suma` gem

**No custom frontend code exists.** The site's UI comes entirely from the concept-browser package. The only local script is `scripts/generate-favicons.mjs`.

## CI/CD

- **`.github/workflows/deploy.yml`** — Builds and deploys to GitHub Pages on push to `main`. Uses `concept-browser build` (not raw `vite`). Uses committed `.datasets/` data (no private repo access).
- **`.github/workflows/regenerate-concepts.yml`** — Checks out the private `iso-10303` repo, runs `rake concepts:all` via the `suma` gem, and opens a PR with updated `.datasets/`. Runs weekly Mondays or on manual trigger. Uses `METANORMA_CI_PAT_TOKEN` secret for private repo access.

## Concept YAML Format

Each concept is a YAML file in `.datasets/<dataset>/concepts/` named `<schema>.<entity>.yaml`. Concepts follow the Glossarist schema (version 3) with fields for term designations, definitions, sources, and relationships. Inline references use `{{urn:...,display text}}` for URN links and `{{term,id}}` for concept cross-references.
